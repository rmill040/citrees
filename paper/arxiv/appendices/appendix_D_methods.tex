\section{Algorithmic and reproducibility notes}
\label{app:methods}

This appendix records algorithm and reproducibility details needed to interpret the main
results. Section~\ref{sec:method} gives the tree-growing loop, the setup and
proof appendices give the fixed-node results, and
Appendix~\ref{app:complexity-details} gives runtime and memory accounting for
the benchmarked tree-growing algorithm.

\subsection{Scope relative to the theory}

The theory covers a narrow setting: exhaustive fixed-$B$ Stage~A feature
selection at a fixed node under the complete permutation null, with a
label-independent tested feature set and resample budget. The benchmarked
tree-growing algorithm includes several practical settings that are useful
but fall outside that theorem:
\begin{itemize}
  \item early stopping,
  \item feature and threshold candidate ordering inside early stopping,
  \item feature muting,
  \item bounded threshold sets,
  \item bootstrap resampling and forest aggregation,
  \item optional honesty.
\end{itemize}
The fixed-node result applies only to exhaustive fixed-$B$ Stage~A $p$-values for
that label-independent tested feature set. Other Stage~A and split scores reported
for adaptive tree growing are tree-construction scores.

\subsection{Node-level choices}

\paragraph{Stage~A.}
At each node, the tree-growing rule removes features that are constant on
the node before any \texttt{max\_features} subsampling. If permutation testing
is enabled, Stage~A computes $+1$ Monte Carlo $p$-values, can order candidates by
a preliminary score before testing, and can terminate once a sufficiently strong
candidate has been found. When feature muting is
enabled, features with non-rejecting Stage~A $p$-values can be removed from
descendant candidate feature pools within the same subtree. These choices reduce
computation, but they also make the returned feature field
$q_t^{\mathrm{feat}}$ a
tree-construction search score rather than a full Stage~A $p$-value over the
complete candidate feature pool.

\paragraph{Stage~B.}
Given the selected feature, the tree-growing rule builds threshold
candidates from midpoints between consecutive unique feature values. Exact search uses all such
midpoints. Random sampling chooses a bounded subset of midpoints; percentile and
histogram rules choose bounded representative thresholds derived from the
midpoint distribution, with a fallback to the full midpoint set when the
selected feature has at most four unique values. When sequential stopping is enabled, threshold scanning may then reorder
the candidate thresholds before testing. Because Stage~B is run after Stage~A has
selected a feature using the node response, its returned quantity
$q_t^{\mathrm{split}}$ should be interpreted as a split-selection score.

\paragraph{Split statistics.}
For permutation split testing, Stage~B uses the unweighted sum
of child impurities as the tested statistic and a left tail convention. The
deterministic split search and the final
\texttt{min\_impurity\_decrease} filter instead use the usual weighted impurity
decrease. This distinction matters only for interpreting the Stage~B score; it
does not change the main benchmark comparisons, which are all based on the final
rankings produced by the fitted learners.

\subsection{Permutation budgets and early stopping}

The benchmarked tree-growing rule uses Phipson--Smyth $+1$ Monte Carlo permutation $p$-values
\citep{PhipsonSmyth2010PermutationPValues} for both stages: right tail in
Stage~A and left tail in Stage~B. The library exposes integer resampling budgets
and three presets:
\begin{itemize}
  \item \texttt{minimum}: the smallest budget that makes rejection attainable on
  the discrete $+1$ $p$-value scale,
  \item \texttt{auto}: a larger budget intended to stabilize Monte Carlo error
  near the decision threshold,
  \item \texttt{maximum}: a conservative high-precision setting.
\end{itemize}
When Bonferroni is enabled, the effective threshold depends on the number of
tests at the node, so the realized permutation budget at each node increases with
search width.

\paragraph{Sequential stopping.}
The benchmarked defaults use adaptive sequential stopping. The sequential test
tracks exceedances over permutation draws and checks a Beta posterior for the
exceedance rate at routine-specific points: each draw for Stage~B and batches
for batched Stage~A selector paths. It terminates when the posterior places at
least the configured confidence probability on one side of the nodewise
threshold. A simpler futility/significance rule is also available. Both change
the stopping time: once a stage stops at a data-dependent time, the returned
value is no longer a classical fixed-$B$ Monte Carlo $p$-value. This is why the
main paper separates the fixed-node guarantee from the runtime ablations.

\subsection{Forests, resampling, and honesty}

CIF averages predictions from bootstrap trees. These forests are empirical
extensions of the Stage~A rule, not settings covered by the fixed-node result
itself. The classification tree and forest grids also support
class-conditional bootstrap variants, and both trees and forests optionally
support honest estimation via sample splitting. These settings can be useful in
practice, but they change the training distribution or the estimation target;
they are empirical design choices and are not covered by the theorem.

\subsection{Benchmark output availability}

\Cref{sec:experiments-protocol,sec:experiments-repro} define the benchmark
protocol and configuration-selection rule. This appendix only records output
availability details needed to interpret the appendix tables.
The analysis uses completed benchmark outputs without imputation. Some benchmark
outputs are unavailable, and some datasets have fewer than five standard values
of $k$ because values greater than $p$ are not evaluated.
Summaries over individual results omit unavailable outputs. Analyses that
compare all methods retain only datasets with every method, downstream learner,
and standard value of $k$ present. These
rules explain why ctree and cforest classification comparisons use 22 rather
than 23 datasets and why analyses requiring every method, downstream learner,
and standard $k$ value use 14 classification and 6 regression datasets.
\Cref{tab:output-skips} lists the classification seed skips and the
dexter method exclusion; the regression count comes from the same availability
rule, not from additional rows in this table.

\subsection{Randomness and reproducibility}

Runs are deterministic given the same base seed. Pure Python
permutation routines use local \texttt{numpy} generators, while compiled
parallel routines use seeded streams per resample because generator RNGs
are not uniformly supported in those contexts. This affects how random streams
are constructed in code, but not the substantive reading of the results: the
benchmark is reproducible from the saved artifacts and the reported rankings are
deterministic given the stored configuration and seed schedule.
