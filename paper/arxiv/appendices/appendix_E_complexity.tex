\section{Runtime and memory accounting}
\label{app:complexity-details}

This appendix supports the runtime ablations in \cref{sec:results-practical}.
It gives runtime accounting for the benchmarked tree-growing rule. The
accounting treats training cost as a sum over the nodes reached,
the scans and copies performed, and the candidate tests actually run. It is not
well described by a single product
such as depth times feature count times permutation budget, because candidate
feature pools, node sample sizes, threshold counts, and stopping times all vary across
the tree.

At node $t$, let $n_t$ be the number of samples reaching the node and let $p$ be
the original number of columns in the dense design matrix. Let $a_t$ be the
number of features available to the node before constant-feature pruning, and let
$d_t^{\mathrm{const}} \le a_t$ be the number of those features removed because
they are constant on the node samples. After this pruning and any
\texttt{max\_features} subsampling, let $m_t$ be the number of features sent to
Stage~A. Let $d_t^{\mathrm{mute}}$ be the number of tested Stage~A features
removed from the descendant candidate feature pool by \texttt{feature\_muting}; if feature
muting is disabled, $d_t^{\mathrm{mute}}=0$.

Stage~B is reached only when Stage~A's split flag is true: either the Stage~A
permutation feature test rejects or selector permutation testing is disabled. If
Stage~B is reached, let $\tilde{q}_t$ be the number of candidate thresholds
generated before \texttt{min\_samples\_leaf} filtering, and let
$q_t \le \tilde{q}_t$ be the number retained after filtering.

Let $C^{\mathrm{sel}}_j(n_t)$ denote the cost of the Stage~A statistic used for
feature $j$ at node size $n_t$. In multi-selector mode this is the composite
max-T statistic cost for one observed or permuted draw. Let
$C^{\mathrm{split}}(n_t)$ denote the cost of one Stage~B split-quality statistic
for a fixed threshold, excluding the label shuffle/copy overhead written below.
For the default MC/Gini classification path and PC/MSE regression path these
statistic evaluations are linear in $n_t$ for a fixed number of classes. RDC is
$O(n_t\log n_t)$ for fixed random-projection and class counts. Distance
correlation and mutual information should be read through
$C^{\mathrm{sel}}_j(n_t)$: the current distance-correlation path can use a fast
$O(n_t\log n_t)$ method, but fallback paths can be $O(n_t^2)$, and
mutual information is delegated to scikit-learn.

\paragraph{Per-node accounting.}
The tree-growing algorithm first checks the node-level stopping conditions, including
whether all labels at the node are equal, and computes a leaf value if the node
stops immediately. This contributes $O(n_t)$ work at every realized node. If the
node is eligible for splitting, the algorithm scans all currently available
features and removes features that are constant on the node. This costs
$O(a_t n_t)$ for the data scan. Updating the feature-index array costs another
$O(a_t d_t^{\mathrm{const}})$ for constant-feature removals in the current
dense-data path, with worst case $O(a_t^2)$ if many available features are
constant at the same node. This pruning occurs before \texttt{max\_features}, so
\texttt{max\_features} caps Stage~A width but not the initial constant-feature
scan.

Let $\mathcal{J}_t$ be the set of Stage~A features that enter the main selection
loop and whose score or permutation test is actually evaluated before any
feature-level early stop. This set does not include the optional feature-scanning
prescan, which is accounted for separately. Let $b^{\mathrm{sel}}_{t,j}$ be the
number of permutation draws actually used for feature $j$; when permutation
testing is disabled, take $b^{\mathrm{sel}}_{t,j}=0$. The Stage~A testing work
is
\begin{equation}
  S^{\mathrm{A}}_t
  =
  O\!\left(
    \sum_{j \in \mathcal{J}_t}
      \bigl(b^{\mathrm{sel}}_{t,j}+1\bigr)
      \bigl(C^{\mathrm{sel}}_j(n_t)+n_t\bigr)
  \right),
\end{equation}
where the $+1$ accounts for the observed statistic and the $n_t$ term accounts
for label copies or shuffles inside the permutation test. Feature scanning,
when enabled with selector early stopping, computes prescan scores and sorts
candidates, adding
\begin{equation}
  G^{\mathrm{A}}_t
  =
  O\!\left(
    \sum_{j=1}^{m_t} C^{\mathrm{scan}}_j(n_t)
    + m_t \log m_t
  \right),
\end{equation}
where $C^{\mathrm{scan}}_j$ is the selector-score cost used for ordering. If
\texttt{feature\_muting} removes tested features with non-rejecting Stage~A
p-values, updating the descendant feature-index array adds
$O(a_t d_t^{\mathrm{mute}})$ current-node bookkeeping work.

Stage~B candidate construction begins with unique-value and threshold-generation
work on the selected feature. A safe upper bound for construction and leaf-size
filtering is $O(n_t\log n_t + \tilde{q}_t\log n_t)$, including unique/sort work
and the \texttt{searchsorted} calls used for \texttt{min\_samples\_leaf}
filtering. Bounded non-exact threshold methods reduce $\tilde{q}_t$ up to
method-specific constants: \texttt{random} and \texttt{percentile} return at
most \texttt{max\_thresholds} candidates before filtering, while
\texttt{histogram} can return up to \texttt{max\_thresholds}$+1$ bin edges. The
\texttt{exact} method and the small-\texttt{n\_unique} fallback use all
midpoints. All paths still pay threshold-construction and filtering work.

Let $\mathcal{K}_t$ be the set of retained Stage~B thresholds that enter the
main split-selection loop and whose permutation test or non-permutation split
score is actually evaluated before any threshold-level early stop. Let
$b^{\mathrm{split}}_{t,k}$ be the corresponding realized permutation count. The
Stage~B testing work is
\begin{equation}
  S^{\mathrm{B}}_t
  =
  O\!\left(
    \sum_{k \in \mathcal{K}_t}
      \bigl(b^{\mathrm{split}}_{t,k}+1\bigr)
      \bigl(C^{\mathrm{split}}(n_t)+n_t\bigr)
  \right).
\end{equation}
Threshold scanning, when enabled with splitter early stopping, uses weighted
child impurity as an ordering score and adds
$O(q_t C^{\mathrm{split}}(n_t) + q_t\log q_t)$. This ordering score is separate
from the Stage~B p-value statistic.

Finally, if the Stage~B split is accepted and the impurity decrease is large
enough, the current dense-data path materializes left and right child matrices with
all $p$ columns. This data copy costs $O(n_t p)$ at that split. Nodes that reach
Stage~A but become leaves because Stage~A or Stage~B does not reject still incur
the work already described. Non-internal leaves can also arise after
constant-feature pruning leaves no available feature, after threshold generation
and \texttt{min\_samples\_leaf} filtering leave no valid threshold, or after an
accepted threshold fails the \texttt{min\_impurity\_decrease} gate. Such leaves
contribute the costs incurred up to the last reached code path, but they do not
allocate child matrices.

\paragraph{Fixed-$B$ and adaptive modes.}
In the exhaustive fixed-$B$ reference setting, the realized Stage~A set
$\mathcal{J}_t$ is the full tested feature set, the realized Stage~B set
$\mathcal{K}_t$ is the full retained threshold set, and the realized permutation
counts are the fixed budgets. When permutation testing is enabled, adaptive
stopping can reduce realized work by shrinking the sets
$\mathcal{J}_t,\mathcal{K}_t$ and the realized counts
$b^{\mathrm{sel}}_{t,j}, b^{\mathrm{split}}_{t,k}$. Feature and threshold
scanning are ordering heuristics: they spend prescan work in order to present
promising candidates earlier to the stopping rule. They can reduce realized
runtime when the ordering shortens the stopping path enough to offset that
prescan work, but they do not improve the worst-case bound because the stopping
rule can still examine all candidates and use the full budget.

\paragraph{From nodes to trees and forests.}
Let $T_t$ denote the applicable node-local cost for node $t$: immediate-stop
work, Stage~A work, Stage~B work, and child-copy work as far as the recursive
builder reaches. Tree training cost is the sum over all realized tree nodes, not
only over internal split nodes:
\begin{equation}
  T_{\mathrm{tree}} = \sum_{t \in \mathcal{V}(\mathcal{T})} T_t.
\end{equation}
Immediate-stop leaves contribute only the node-level stopping and value
computation term, while leaves that fail after Stage~A or Stage~B contribute the
corresponding testing terms. For a binary tree,
$|\mathcal{V}(\mathcal{T})|=2|\mathcal{I}(\mathcal{T})|+1$, and a depth cap
gives the usual exponential crude bound on node count, but $n_t$, $a_t$, $m_t$,
$\tilde{q}_t$, $q_t$, and the realized permutation counts usually change
substantially with depth.

For a forest of $M$ trees, let $s_r$ be the realized sample count passed to tree
$r$ after bootstrap and any classifier sampling method. The forest fit first
validates the full input and then fits the trees. The forest fit can be
summarized as
\begin{equation}
  T_{\mathrm{forest}}
  =
  O(np)
  + \sum_{r=1}^{M}
      \left(C^{\mathrm{sample}}_r + O(s_r p) + T_{\mathrm{tree},r}\right)
  + O(Mp)
  + \mathbf{1}_{\mathrm{oob}} T_{\mathrm{oob}},
\end{equation}
where $O(s_r p)$ accounts for the per-tree training data supplied to tree $r$.
With \texttt{bootstrap=False}, the realized root size is $n$ and
\texttt{max\_samples} is unused; with bootstrap sampling, \texttt{max\_samples}
is a requested sample size or cap, and classifier sampling methods can make the
realized $s_r$ smaller. If honesty is enabled, the tree-structure root size is
the splitting subsample size, and training also pays for the split/estimation
partition plus a leaf re-estimation pass. Optional OOB scoring is additional
post-fit work, not part of the tree-building sum.

\paragraph{How runtime hyperparameters enter the accounting.}
\begin{itemize}
  \item \textbf{\texttt{max\_features}.} This caps $m_t$ and therefore the
  Stage~A testing width. It does not cap the constant-feature preprocessing pass
  over all $a_t$ available features.
  \item \textbf{\texttt{feature\_muting}.} This can shrink descendant values of
  $a_t$ and $m_t$. It can also add $O(a_t d_t^{\mathrm{mute}})$ current-node
  feature-index bookkeeping when tested features with non-rejecting Stage~A
  p-values are removed.
  \item \textbf{\texttt{threshold\_method} and \texttt{max\_thresholds}.}
  Non-exact threshold methods use \texttt{max\_thresholds} to reduce
  $\tilde{q}_t$ up to method-specific constants; \texttt{exact} and the
  small-\texttt{n\_unique} fallback use all midpoints.
  \item \textbf{\texttt{early\_stopping} settings, \texttt{feature\_scanning},
  and \texttt{threshold\_scanning}.} With permutation testing enabled, these are
  realized-work optimizations. They change candidate order and may reduce the
  number of candidates and permutations actually evaluated when the stopping
  path benefits from that ordering, but the worst case remains the exhaustive
  candidate feature pool and full permutation budget. In the collected ablations,
  adaptive stopping accounts for the largest observed CIF runtime change, while
  scanning alone does not show comparably clear separate timing gains.
  \item \textbf{Bonferroni correction and resampling presets.} When Bonferroni
  correction is enabled, the nodewise threshold is divided by the number of
  tests at that stage. Writing $r$ for that test count, string-valued resampling
  presets recompute the permutation budget at this stricter threshold: the
  \texttt{minimum} budget scales like $r/\alpha$, the \texttt{maximum} budget
  like $r^2/\alpha^2$, and \texttt{auto} uses the same stricter threshold in its
  normal-quantile formula. For an explicit integer budget, the budget is
  multiplied by $r$.
  \item \textbf{\texttt{max\_samples}, \texttt{n\_estimators}, and
  \texttt{n\_jobs}.} These affect forest scaling in different ways:
  \texttt{max\_samples} affects the requested or capped bootstrap sample size,
  \texttt{n\_estimators} sets the number of tree fits, and \texttt{n\_jobs}
  changes tree-fit scheduling and memory pressure.
\end{itemize}

\paragraph{Memory accounting.}
At a node, feature-index bookkeeping and scanning arrays use $O(a_t+m_t)$ space,
and threshold construction uses $O(n_t+\tilde{q}_t+q_t)$ auxiliary space.
Permutation tests add statistic, mask, label, and selector-specific workspaces.
The dense-data terms dominate in many fits: validation materializes a dense
$O(np)$ design matrix, and each accepted split materializes dense left and right
child matrices with combined size $O(n_t p)$. Peak single-tree training memory
is therefore bounded by the validated root matrix plus dense child arrays along
the active recursion path and sibling arrays retained by parent frames. Forest
peak training memory also grows with the number of concurrently fitted trees and
their per-tree training data. Persistent fitted storage is the sum of the stored
node records over trees, plus forest-level feature-importance and estimator
arrays.
