\section{Algorithmic and reproducibility notes}
\label{app:methods}

This appendix keeps only the implementation details that materially affect how
the main results should be read. Section~\ref{sec:method} gives the algorithmic
loop, Appendices~A--C give the fixed-node proofs, and Appendix~\ref{app:complexity-details}
gives the detailed complexity formulas.

\subsection{Scope relative to the theory}

The theorem-backed object is narrow: exhaustive fixed-$B$ Stage~A feature selection at a
pre-specified node under the complete permutation null. The benchmarked learner
adds several practical controls that are useful algorithmically but fall outside
that theorem package:
\begin{itemize}
  \item early stopping,
  \item feature and threshold candidate ordering inside early stopping,
  \item feature muting,
  \item approximate threshold sets,
  \item bootstrap resampling and forest aggregation,
  \item optional honesty.
\end{itemize}
Accordingly, only the fixed-node exhaustive Stage~A p-values retain the
calibrated Monte Carlo interpretation from \cref{sec:theory}. The remaining
reported Stage~A or split scores are algorithmic quantities.

\subsection{Node-level implementation choices}

\paragraph{Stage~A.}
At each node, \citrees removes features that are constant on the node before any
\texttt{max\_features} subsampling. If permutation testing is enabled, Stage~A
computes $+1$ Monte Carlo p-values, can order candidates before testing so the
sequential stopping rule sees promising features earlier, and may stop early
once a sufficiently strong candidate has been found. When feature muting is
enabled, features with non-significant Stage~A scores can be removed from
descendant candidate sets within the same subtree. These choices reduce
computation, but they also make the returned score $s_t^\star$ an algorithmic
Stage~A score rather than an exhaustive fixed-family p-value.

\paragraph{Stage~B.}
Given the selected feature, \citrees builds threshold candidates from midpoints
between consecutive unique feature values. Exact search uses all such
midpoints; random, percentile, and histogram rules choose the threshold set
itself, with a small-sample fallback to the full midpoint set. When sequential
stopping is enabled, threshold scanning may then reorder that chosen set before
testing. Because Stage~B is run only after choosing a feature with the same
labels, its returned quantity $s_t^{\mathrm{split}}$ is read as a post-selection
split score rather than as a calibrated p-value.

\paragraph{Split statistics.}
For permutation-based split testing, the implementation uses the unweighted sum
of child impurities as the tested statistic and a left-tail convention. The
deterministic split search and the final
\texttt{min\_impurity\_decrease} filter instead use the usual weighted impurity
decrease. This distinction matters only for interpreting the Stage~B score; it
does not change the main paper's benchmark summaries, which are all based on the
final rankings produced by the fitted learners.

\subsection{Permutation budgets and early stopping}

\citrees uses Phipson--Smyth $+1$ Monte Carlo permutation p-values
\citep{PhipsonSmyth2010PermutationPValues} for both stages: right-tail in
Stage~A and left-tail in Stage~B. The library exposes integer resampling budgets
and three presets:
\begin{itemize}
  \item \texttt{minimum}: the smallest budget that makes rejection attainable on
  the discrete $+1$ p-value grid,
  \item \texttt{auto}: a larger budget intended to stabilize Monte Carlo error
  near the decision threshold,
  \item \texttt{maximum}: a conservative high-precision setting.
\end{itemize}
When Bonferroni is enabled, the effective threshold depends on the number of
tests at the node, so the realized per-node permutation budget increases with
search width.

\paragraph{Sequential stopping.}
The benchmarked defaults use adaptive sequential stopping. The implementation
tracks exceedances inside batches and stops once a Beta--Binomial posterior is
confident that the latent exceedance rate is either below or above the nodewise
threshold. A simpler futility/significance rule is also available. Both are
practical heuristics: once a stage stops at a data-dependent time, the returned
value is no longer a classical fixed-$B$ Monte Carlo p-value. This is exactly
why the main paper separates the fixed-node guarantee from the empirical
practical-controls result.

\subsection{Forests, resampling, and honesty}

CIF averages predictions from bootstrap trees. These forests are empirical
extensions of the theorem-backed Stage~A rule, not objects covered by the
fixed-node result itself. The classification implementation also supports
class-conditional bootstrap variants, and both trees and forests optionally
support honest estimation via sample splitting. These settings are useful
algorithmically, but they change the training distribution or the estimation
target and should therefore be read as empirical design choices rather than as
parts of the theorem package.

\subsection{Benchmark protocol summary}

All benchmark methods are treated as feature rankers. A method first produces a
ranking on the training split; downstream models are then fit on the top-$k$
selected features and evaluated on held-out data. The real-data benchmark uses
5-fold cross-validation over 5 random seeds, standard budgets
$k \in \{5,10,25,50,100\}$, and larger endpoint budgets only for the separate
high-$p$ saturation check. If a method family has multiple configurations, the
analysis uses one selected configuration per task, chosen by mean score on the
same benchmark surface reported in the paper. The resulting tables are therefore
descriptive post-selection summaries rather than nested external validation
estimates.

\subsection{Randomness and reproducibility}

The implementation is deterministic given the same base seed. Pure Python
permutation routines use local \texttt{numpy} generators, while compiled
parallel routines use per-resample seeded streams because generator-based RNGs
are not uniformly supported in those contexts. This affects how random streams
are constructed in code, but not the substantive reading of the results: the
benchmark is reproducible from the saved artifacts and the reported rankings are
deterministic given the stored configuration and seed schedule.
