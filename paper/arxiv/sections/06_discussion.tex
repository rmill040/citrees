\section{Discussion}
\label{sec:discussion}

The clearest use case for CIF in these experiments is top-$k$ feature selection
for a downstream model. The benchmark asks whether a ranking places useful
features early enough for fixed downstream learners, and the strongest evidence
is in classification. The regression benchmark points in the same direction, but
it includes fewer datasets and has wider intervals. The fixed-node Stage~A
theorem provides the local guarantee; the benchmark evaluates the fitted tree
and forest rankings.

\paragraph{Practical guidance.}
A CIF ranking should be judged through the downstream model that will use the
selected features. In practice, choose $k$ on held-out data over a small grid
that includes the intended feature budget and, when feasible, the full feature
set as a sanity check. When prediction accuracy is the main target, tree-ensemble
baselines such as random forests, ExtraTrees, and boosted trees should remain in
the comparison set. When the target is variable screening or a ranked feature
list for later modeling, CIF should also be compared with ctree, cforest, and
forest-based screening methods such as Boruta, permutation or conditional
permutation importance, and RF-RFE
\citep{GenuerPoggiTuleauMalot2010VariableSelectionUsingRandomForests}.

For similar top-$k$ ranking workloads, the CIF runtime ablations, which vary one
setting at a time, support adaptive stopping and histogram-256 thresholding as
practical defaults. Disabling adaptive stopping and replacing histogram-256
thresholding with exact threshold search produce the largest CIF slowdowns,
while \cref{tab:cif-runtime-hyperparams} reports a largest absolute downstream
score change of 0.011. Bonferroni correction should stay in the main setting
because removing it changes the Stage~A testing rule; report it as a sensitivity
analysis when it is varied. The ranking ablations separate the remaining choices.
Reducing CIF to one tree gives the largest downstream score loss; split-count
ranking and feature muting have much smaller effects, and removing bootstrap
sampling is close to neutral in classification but worse in regression.

In sparse high-$p$ settings, candidate feature availability should be monitored
directly. The simulations show that default candidate sampling can leave
informative features out of many forest split decisions, especially in sparse
classification designs. Useful diagnostics include top-$k$ stability across
seeds, the share of splits using the final top-$k$ features or any prespecified
positive control features, the number of distinct low-ranked or known null
features used in splits, and sensitivity to less aggressive candidate sampling
or to a small CIF-all run with fewer seeds or trees.

\paragraph{Limitations.}
The fixed-node result is a reference guarantee for exhaustive fixed-$B$ Stage~A
testing at one node, with the tested features and permutation budget fixed
independently of the node responses. The benchmarked trees and forests add
adaptive stopping, candidate ordering, feature muting, bounded threshold sets,
bootstrap samples, and forest aggregation, so their behavior is assessed through
the ranking benchmark.

The benchmark coverage is uneven across analyses. The ctree and cforest
classification comparisons use 22 datasets, while the regression main table uses
8. Complete-case rank and interval analyses are smaller, with 14 classification
datasets and 6 regression datasets. The regression results therefore require
more caution than the classification results, especially for comparisons with
wide intervals.

The candidate-availability analysis isolates a sparse high-$p$ setting in which
forest candidate sampling can limit informative-feature exposure: wider
candidate sets increase exposure, but they also change runtime and the mix of
noise features used by splits. The point is diagnostic rather than prescriptive:
candidate exposure should be checked alongside downstream performance and
runtime before changing the candidate set.
